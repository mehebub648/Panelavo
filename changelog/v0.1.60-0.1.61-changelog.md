# v0.1.60 → v0.1.61

## Fix: Compose operations failed for sites using named volumes ("volume must be a mapping")

Any rootful Compose site whose configuration mismatches CloudPanel's expected port — so Panelavo publishes a loopback-remapped ephemeral config — failed at the **Validate configuration** step with:

```
services.<service>.volumes.0.volume must be a mapping
```

whenever a service used a named volume (e.g. `postgres-data:/var/lib/postgresql/data`). This blocked compose up/deploy/restart/validate for a very common setup (Postgres, Redis, any persistent named volume).

### Root cause

`remapResolvedCompose()` takes `docker compose config --format json`, decodes it with associative `json_decode(..., true)`, remaps published ports onto `127.0.0.1`, and re-encodes it as the ephemeral config Compose then re-loads. Compose emits long-syntax mounts with an options sub-key that is an **empty object** when no options are set — a named volume resolves to `{type: volume, source, target, volume: {}}`. PHP's associative decode collapses `{}` to `[]`, and `json_encode` renders that back as a JSON array. Compose rejects `"volume": []` because the sub-key must be a mapping.

This is the same empty-object-to-empty-array hazard the function already handled for `networks.*.ipam`; it simply wasn't applied to mount option sub-keys.

### Fix

`remapResolvedCompose()` now strips synthetic empty long-syntax mount option sub-keys (`volume`, `bind`, `tmpfs`, `cluster`) — absent means the same defaults — while preserving any operator-set options (e.g. `bind: {create_host_path: true}`). Non-empty option maps encode correctly on their own; only the empty `[]`-corrupted shapes were the problem.

Verified against the real failing site: the buggy round-trip reproduces the exact error, and the fixed config passes `docker compose config --quiet`.

### Files

- `scripts/cloudpanel-bridge.php`: strip empty mount option sub-keys in `remapResolvedCompose()`; extended the `--self-test-ports` self-test with a service that uses both an optionless named volume and a bind mount with options, asserting the empty option is dropped, real options survive, no `"volume":[]` is encoded, and the source config is not mutated.

### Deployment note

The remapping logic runs in the root-owned broker (`/usr/local/libexec/panelavo/cloudpanel-bridge.php`), which the in-panel self-update does not replace. Already-installed servers need the updated broker bridge reinstalled as root (re-run `sudo bash setup.sh`, or reinstall that one file) before the fix takes effect. New installs pick it up automatically from `setup.sh`.
