# 0.1.23 to 0.1.24

- Add an **Environment** manager to Site Settings: auto-detects `.env`, `.env.local`, and `.env.production` in the application root, edits them with comment-preserving rewrites, and keeps `.env` in sync with a marker-tagged export block in the site user's `~/.profile`. PM2 launches from Operations now also inject the site's `.env` variables (CloudPanel's expected `PORT` always wins), so applications receive their configuration even when they never parse `.env`. Environment values are only rendered for users who can manage the website.
- Create the destination folder automatically (recursively, site-user-owned) when extracting archives in the File Manager, and resolve `.`/`..` destinations client-side so they work from any folder.
- Add a **Terminal** tab per website: bounded command execution strictly as the site user through a login shell (managed environment included), with persistent working directory, command history, and instructions for connecting an external SSH/SFTP terminal.
- Add a **Runtime** panel to Operations: auto-detected live state for PM2 processes (status, CPU, memory, restarts, uptime, PID), Docker Compose containers (state, health, published ports), site-owned listening ports, and a `.env` drift check that compares configured variables against the running process environment (key names and verdicts only — values never leave the server).
- Validate the new `env` and `terminal` section requests with strict shared schemas before they reach the bridge, and add regression coverage plus a `--self-test-env` bridge self-test for dotenv parsing and rewriting.
