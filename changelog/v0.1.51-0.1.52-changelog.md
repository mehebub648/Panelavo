# v0.1.51 → v0.1.52

## Dedicated database manager replaces CloudPanel's port-8443 phpMyAdmin links

The Databases section used to link to `https://<panel-host>:8443/pma/...` —
CloudPanel's own portal, which serves a self-signed certificate and is
deliberately firewalled by `setup.sh`. Browsers refused the connection with
`NET::ERR_CERT_AUTHORITY_INVALID` and, because CloudPanel sends HSTS, offered
no bypass.

- `setup.sh` now provisions a standalone phpMyAdmin as its own CloudPanel PHP
  site on `database.<ip>.<base-domain>` (override with `DB_MANAGER_DOMAIN`).
  The domain is covered by the same wildcard DNS record as the panel, so it
  receives a real Let's Encrypt certificate in the SSL step. Provisioning is
  idempotent and non-fatal: a failure only warns and hides the panel's
  database links.
- phpMyAdmin is configured for cookie authentication against local MySQL
  only: no root logins, no arbitrary servers, `AllowNoPassword` off. Users
  sign in with the database user's own credentials, so MySQL itself enforces
  per-site scope. Port 8443 stays firewalled.
- `setup.sh` records the provisioned address in `DATABASE_MANAGER_URL` in the
  panel's `.env.local` (appended on re-runs for existing installs). The panel
  falls back to deriving `https://database.<ip>.<base-domain>` when the
  variable is missing.
- The Databases section's "Open phpMyAdmin" and per-database "Manage" buttons
  now target the database manager (`?db=` preselects the database after
  login). The buttons are hidden when no manager address can be resolved.
- The database manager's CloudPanel site is panel infrastructure: like the
  panel's own site, it is filtered from site listings and protected from
  per-site operations (`src/server/sites/database-manager.ts`).

Existing installs: re-run `sudo bash setup.sh` from the repo root to
provision the database manager and record `DATABASE_MANAGER_URL`.
