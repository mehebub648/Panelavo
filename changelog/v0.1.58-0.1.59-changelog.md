# v0.1.58 → v0.1.59

## Auto-detect Compose files kept in a subfolder

Docker Compose preflight and every Compose operation now discover a Compose file that lives in a subfolder of the application root (for example `docker/compose.yaml`), instead of only recognizing one placed directly in the root.

### What changed

- Added a single `findComposeFile()` helper in `scripts/cloudpanel-bridge.php` and routed all Compose-file discovery through it (framework detection, the Operations state snapshot, rootless-migration analysis and preparation, and the loopback port-binding fix). Previously each of these repeated an identical root-only, four-candidate loop.
- The application root remains the canonical location and a root-level Compose file always wins. Only when no root candidate exists does the helper run a bounded (max depth 3), breadth-first, alphabetically deterministic scan of subdirectories, skipping dependency and VCS trees (`node_modules`, `vendor`, `.git`, `.svn`, `.hg`, `backups`, `storage`, `cache`, `.cache`, `tmp`).
- The returned path stays relative to the application root (e.g. `docker/compose.yaml`). Compose commands already run with the application root as their working directory, so `docker compose -f <relative>` and the host-safety scan keep resolving against that root. The Compose project directory defaults to the file's own folder, matching native `docker compose` behavior.

### Behavior notes

- No new configuration. Detection is automatic; a root Compose file is unchanged in every respect.
- Updated the "no Compose file" preflight remediation copy and synchronized `README.md` and `DEPLOYMENT.md` to describe subfolder discovery.
