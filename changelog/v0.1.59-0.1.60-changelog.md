# v0.1.59 → v0.1.60

## Site-write users can self-initialize their own rootless Docker runtime

Previously, bringing up a site user's private rootless Docker daemon was a single Super-Admin-only host fix that also ran `apt-get install` and `usermod` as root. That meant only a Super Admin could get a site's rootless Docker working, and it couldn't safely be delegated (running root package management on behalf of a tenant is effectively granting root).

This release splits rootless bring-up into two clean boundaries so any site-write user can start their own runtime without a Super Admin, while host software changes stay privileged.

### Split of responsibilities

- **Host provisioning — Super-Admin-only, and now done by `setup.sh`.** Installing the shared rootless packages (Docker Engine, CLI, Buildx, Compose v2, `docker-ce-rootless-extras`, `fuse-overlayfs`, plus the existing `uidmap`/`dbus-user-session`/`slirp4netns`) and allocating subordinate UID/GID ranges remain a root boundary. `setup.sh` now provisions all of this from Docker's official APT repository on every install, so a fresh VPS is ready out of the box.
- **Per-user runtime self-init — NEW, site-write self-service.** A new `initialize-rootless-runtime` fix (`scope: "site"`) starts only the requesting site user's own daemon: enable their linger, start their user manager, run `dockerd-rootless-setuptool.sh` as the user, and enable `docker.service`. It **never** runs `apt-get` or `usermod`, and refuses — without mutating anything — when the host is not yet provisioned, pointing the operator to a Super Admin.

### Behavior

- The **Rootless prerequisites** preflight check now reflects *host* provisioning (packages + subordinate ranges), independent of whether the per-user daemon is running, using a new daemon-independent Buildx-plugin probe.
- The **Site user service manager** and **Private rootless daemon** checks now offer the site-scoped runtime self-init fix when the host is provisioned, and fall back to the Super-Admin host fix when it is not.
- Once the host is provisioned, a CloudPanel admin, site manager, or panel overlay admin can start their site's rootless runtime with no Super Admin involvement.

### Security

Letting site users trigger the *host* package installs was explicitly rejected: `apt-get install` runs maintainer scripts as root and would grant any site-write tenant effective root over shared infrastructure. The split keeps every host mutation Super-Admin-only; the self-service path is limited to actions that affect only the requesting site user's own runtime — the same rationale that already lets site-write users run ordinary Compose lifecycle actions.

### Files

- `scripts/cloudpanel-bridge.php`: new `rootlessHostProvisioned()` / `dockerBuildxPluginInstalled()` helpers and `hostRootlessReady`/`buildxHostReady` capability fields; refactored the per-user bring-up into a shared `bringUpRootlessUserDaemon()`; added `initializeRootlessRuntime()` and its `executeFix` dispatch; authorized `initialize-rootless-runtime` for site-write users (per-site lock) while keeping every other fix Super-Admin-only and host-serialized.
- `src/server/sites/site-operations.ts`, `src/types/operations.ts`, `src/schemas/operations.ts`: new fix, scope-based authorization, host-vs-runtime fix selection, and capability types.
- `setup.sh`: rootless Docker host provisioning on every install.
- Docs (`AGENTS.md`, `ARCHITECTURE.md`, `DEPLOYMENT.md`, `README.md`) and tests updated.

### Deployment note

The detection/authorization logic runs in the root-owned broker (`/usr/local/libexec/panelavo/cloudpanel-bridge.php`), which the in-panel self-update does not replace. Already-installed servers need the updated broker bridge reinstalled as root (re-run `sudo bash setup.sh`, or reinstall that one file) before the new behavior takes effect.
