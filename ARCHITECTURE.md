# Architecture

Panelavo is a single Next.js App Router application deployed as a CloudPanel Node.js site. Nginx/CloudPanel terminates public TLS and proxies the panel domain to the single PM2-managed Next.js process bound only to `127.0.0.1:10443`. The installer gives the public proxy a 1,900-second read/send timeout so synchronous Operations and Backups requests can complete within the application's roughly 30-minute bound. Port 10443 is not a public authentication surface; emergency direct access uses an SSH tunnel.

Browser requests enter pages and route handlers in `src/app`. Shared validation lives in `src/schemas`; UI components live in `src/components`. Protected server routes obtain opaque application sessions from `src/server/auth`, revalidate current CloudPanel identity and roles, and call the live adapter in `src/server/cloudpanel`. The adapter sends bounded JSON over stdin to `/usr/local/libexec/panelavo/panelavo-broker`, the only Panelavo-owned sudo command. `setup.sh` installs that broker and its PHP bridge root-owned outside the mutable application tree. The broker validates its protocol and allow-listed operations before using CloudPanel's CLI or Symfony/Doctrine internals. CloudPanel remains authoritative for users, roles, MFA, sites, assignments, site types, runtimes, roots, and upstream settings.

The bridge also executes allow-listed Git operations as each site's Unix user. Git is non-interactive, uses that user's home and deployment key, records new SSH host keys with OpenSSH's `accept-new` policy, and is bounded by a five-minute timeout. Clones are restricted to empty configured site roots so existing application files cannot be overwritten, with one exception: an ACME-managed `.well-known` directory is preserved while the repository is checked out. Working-tree paths come from NUL-delimited Git status output and are revalidated against live status before per-file diff or discard operations. Diff content is loaded only when a file is selected. Pulls preserve dirty tracked and untracked work through an automatic stash/restore cycle; a conflicted restore leaves both conflict markers and the safety stash for recovery.

File-manager uploads use base64 JSON and are limited to 64 MiB decoded at both browser and bridge boundaries. The panel Nginx vhost accepts request bodies up to 96 MiB to cover encoding and JSON overhead; `setup.sh` owns that proxy directive and applies it with validation and rollback. Archive extraction may target a folder that does not exist yet: the bridge validates each path segment, rejects traversal and symlinked levels, creates the missing directories owned by the site user, and re-verifies containment before extracting.

Per-site environment management treats the application root's `.env` (plus `.env.local` and `.env.production`) and the site user's login environment as one synchronized surface. The bridge parses and rewrites dotenv files while preserving comments and unknown lines, and mirrors the primary `.env` into a marker-tagged `export` block in `~/.profile` — everything outside the markers is preserved. PM2 launches additionally inject the site's `.env` (filtered to safe key shapes and sizes; reserved variables and `LD_*` are skipped, and CloudPanel's expected `PORT` wins), so a process's live environment matches the configured one without the application parsing `.env`. Environment reads return secret values, so the panel loads them only for users with site-write access; the Operations runtime payload carries key names and sync verdicts only. When Compose reports missing variables, Operations may submit values only for those detected keys through a bounded upsert; the bridge merges them into `.env` without returning or replacing unrelated secrets.

Per-site backups are on-server snapshots the bridge builds under `/home/<site-user>/backups/<id>/`: a gzip tar of the application root archived as the site user, plus a `clpctl db:export` gzip of each selected database (run as root, then chowned to the site user), described by a `manifest.json`. Creation is atomic — any failed step deletes the partial snapshot directory, so a listed backup is always complete — and a retention pass keeps the newest ten by timestamp id. Restore extracts the archive over the live tree (an overlay, not a clean-room replacement) and imports databases only into databases that still belong to the site; a database removed since the backup is skipped because the bridge cannot recreate it without credentials. Backup jobs take the same per-site operations lock as deployments. Snapshots live inside the site home so they are reachable through the File Manager and SFTP without a bespoke large-file download path, and the section is loaded only for users with site-write access.

The per-site Terminal executes one browser-submitted command line as the site's Unix user — deliberately a user-supplied command, unlike Operations' fixed argument arrays, because the boundary is the same unprivileged Unix account the user could reach over SSH. The bridge runs it through `sudo -u <site-user>` in an emptied environment via a non-interactive `bash -l -c` login shell (so the managed profile block applies), with a validated working directory contained in the site home, a three-minute kill timeout, bounded output, and a random per-invocation marker that captures the final working directory so `cd` persists between commands. It never runs as root and grants nothing the site user's SSH session would not.

Project access is a broker-enforced invariant. When an authorized workspace or section is opened, and again before a section mutation, the root-owned bridge verifies that the configured application root has a named POSIX ACL granting its CloudPanel site user `rwX` access and an inheritable default ACL. A missing invariant triggers one physical, non-symlink-following traversal contained under `/home/<site-user>/htdocs`: existing paths receive `rwX` and every directory receives the default entry so files subsequently created by containers or other runtime UIDs remain manageable by Git, backups, the File Manager, Environment manager, and Terminal. Ownership is preserved, execute permission is added only to directories or already-executable files, and no access is granted to other users. `setup.sh` installs the required `acl` package; a missing tool blocks with trusted-installer guidance.

The Super Admin update API reads a configurable public HTTPS Git repository from the panel settings store. Checks use `git ls-remote`; installs launch the non-root `scripts/self-update.sh` worker. The worker holds a `.data` lock, shallow-clones `main`, verifies the Panelavo package contract and declared broker protocol, and refuses the release before build or deployment unless the installed root broker passes its health handshake. It then installs from the frozen pnpm lockfile and completes a production build in staging before synchronizing the live tree. While the persisted state is queued, updating, or reloading, the shared panel layout blocks interaction and authenticated APIs reject normal operations, while server-rendered panel pages remain available behind the overlay. It excludes `.data` and `.env.local`, records durable status and logs, then cleans staging and records the old process PID before asking PM2 to reload. Because PM2 kills the worker with the old process tree, the replacement Panelavo process owns completion of this PID-bound handoff. Legacy states whose installed and remote commits already match are recovered automatically, including removal of their stale update lock. The repository is a trusted application-code source but receives no root execution path; broker upgrades require a separately trusted root `setup.sh` run.

Application-owned metadata, persistent request-throttling counters, and encrypted credentials live under `.data` in the deployed site. Host configuration comes from `.env.local`; secrets must not reach client components. A single process is required by the current session and mutation-serialization design.

## Managed website Operations

The CloudPanel site type, including Panelavo's Docker overlay for CloudPanel reverse-proxy records, is authoritative. `scripts/cloudpanel-bridge.php` inspects only the configured application root and returns structured architecture evidence, manifests, declared scripts, installed tool versions, permissions, PM2 state, Compose capabilities, and validation results. The same report carries a runtime snapshot — PM2 process details, the Compose project's container states/health/published ports, site-owned listening sockets, and a values-free `.env` drift verdict per key — sampled synchronously per request; there is no background polling or streaming. `src/server/sites/site-operations.ts` converts that report into a primary architecture, alternatives, explicit preflight checks, action groups, and a recommended server-owned plan. A manifest can be detected while every related action remains blocked because its runtime, configuration, permission, or safety check failed.

The adapters cover npm, pnpm, Yarn, and Bun lockfile selection; Composer with Laravel and WordPress actions; Python through uv, Poetry, Pipenv, or pip plus Django actions; direct static sites; reverse-proxy checks; PM2 process management; and Docker Compose. The configured root is not searched recursively. A JavaScript workspace therefore needs usable root scripts or explicit root-level configuration, and Panelavo does not guess a generated static output directory or change the CloudPanel document root automatically.

The browser submits only a validated action identifier plus a narrowly validated script or process name when required, or a known deployment-plan identifier. The server expands these into fixed executable argument arrays, a fixed working directory, bounded time and output, and no shell, re-verifying every preflight precondition at execution time. Commands run in an emptied environment (no panel or CloudPanel secrets) whose PATH covers the site user's Node.js, Bun, Composer, and Python tool directories, with Poetry, Pipenv, and pip pinned to project-local virtual environments. A per-site operation lock rejects overlapping runs, plans stop after the first failed step, and the UI keeps missing tools and other blockers visible instead of attempting fallback installation.

Each Docker site has an independent rootless daemon, socket, object namespace, and data store owned by its CloudPanel site user. The bridge injects `HOME`, `USER`, `XDG_RUNTIME_DIR`, `DBUS_SESSION_BUS_ADDRESS`, and the exact `unix:///run/user/<uid>/docker.sock` only into Docker/user-systemd helpers; the ordinary empty-environment runner is unchanged and there is no root-socket fallback. Resolved runtime files are site-owned mode 0600 inside a validated mode-0700 directory under that runtime directory and are deleted after use. Ordinary Compose lifecycle actions use the same site-user/SSH boundary as Terminal and therefore require site-write access, not Super Admin approval. Safety still blocks privileged mode, capabilities, devices, host/shared namespaces, unsafe security options, and out-of-root mounts/build contexts.

Rootless initialization is an allow-listed Super Admin host action. It verifies Docker Engine/CLI, Compose v2, Buildx, rootless extras, functional uidmap helpers, D-Bus, slirp4netns, non-overlapping subordinate ranges, cgroup v2/systemd, storage, linger, user-manager ownership, bus, and socket. It starts the user's systemd manager before running the setup tool and enables the user Docker service. The setup-created rootless context supports interactive SSH without a profile `DOCKER_HOST` override. Per-site stores under `~/.local/share/docker` duplicate shared image layers and are intentionally outside application-root backups; readiness reports their disk/reclaimable use and free space. Userspace port forwarding may change the immediate peer address observed inside containers, so application authorization must use CloudPanel's trusted forwarding configuration.

Migration from legacy rootful Compose is two-phase and Super Admin-only. Preparation performs one cold pull/build per bounded request without creating containers or interrupting traffic, rejects named/external volumes and unsafe or conflicting bind ownership, then stores an expiring root-owned manifest of configuration/image/container identities, subordinate ranges, ports, bind sources, and ownership inventory. Runtime identity resolution uses the running PID directly when it is non-root; for a root init wrapper or privilege-dropping entrypoint, one unique non-root application identity is accepted, while multiple non-root identities remain ambiguous and block migration. Cutover revalidates every input, retains stopped rootful containers, writes a path-safe ownership journal, translates root to the site UID/GID and container id `n >= 1` to `subordinateStart + n - 1`, and reapplies recursive named/default ACLs. It starts the prepared rootless configuration with `--no-build` and verifies state, health, ports, bind access, and HTTP. Failure removes the partial rootless project, restores journaled ownership including newly created mapped files, reapplies ACLs, and restarts/verifies rootful service. A stale journal blocks new migration until explicit recovery succeeds.

The configured CloudPanel upstream/app port is the runtime contract for every port-based site. The bridge compares that expected port with site-owned listening sockets. Node and Python PM2 starts receive `PORT`, `HOST`, and `HOSTNAME` values for the expected loopback endpoint; reverse-proxy sites verify their configured upstream. Recommended plans finish with a bounded HTTP probe and fail instead of claiming success while Nginx would return 502.

Site creation derives an omitted reverse-proxy upstream from the allocated site id as `http://127.0.0.1:<site-id>`. This keeps the CloudPanel vhost and the reserved application port aligned by default while preserving explicit custom upstreams.

Linked services close the loop on reported additional ports: `POST /api/sites/<parent>/services` creates a normal CloudPanel reverse-proxy site targeting `http://127.0.0.1:<port>` and records `parent` plus `serviceName` in the site-meta overlay (`.data/site-meta.json`). CloudPanel stays the source of truth — the grouping is purely panel metadata. The child allocates an id in the parent's category (unique site user and system domain), its generated site-user password is discarded, and the target port is rejected when it is reserved by or serving another website. Service sites expose only settings, domains, vhost, and security sections; operational sections 404 server-side. Deleting a parent returns 409 while children exist, and deletion removes the site's meta so the reserved id/port frees up.

For Compose, Panelavo resolves the configuration to JSON and selects one public entry service using, in order, an existing exact mapping, an explicit `io.panelavo.entrypoint=true` label, a unique port candidate, the dependency graph, or an unambiguous conventional gateway name. The container port comes from `io.panelavo.container-port`, an exact publication, or consistent port/environment/health-check evidence. An unambiguous mismatch is mapped at execution time to `127.0.0.1:<expected-port>` in the private ephemeral configuration; Compose-generated null or empty network IPAM placeholders are removed while real IPAM mappings are preserved, all other published ports are also restricted to loopback, and the source Compose file is not edited. Ambiguity blocks deployment and reports the exact label needed. Additional service ports remain private and are reported so the operator can expose each one through a connected CloudPanel reverse-proxy site belonging to the same application.

Site deletion treats the Unix-account lifetime as load-bearing: the bridge first removes that site's rootless containers, images, volumes, and migration state, stops/disables its user Docker service, removes the private Docker data after explicit destructive confirmation, disables linger, and verifies the manager/socket are gone. Only then may CloudPanel delete the site and Unix user. Other users' rootless daemons and unrelated rootful workloads are outside this cleanup scope.

Blocked host-software checks carry explicit one-click fixes when Panelavo can safely resolve them. Docker/Compose installation, daemon start, and Composer installation remain Super Admin-only, individually confirmed, and host-wide serialized. Violations that require operator judgement — privileged mode, host namespaces, or out-of-root mounts — are never rewritten automatically and retain instructional remediation.

Operations currently run synchronously within one bounded HTTP request and mutate the configured live application root. They do not create release directories, atomically switch releases, or provide automatic code or database rollback. Laravel and Django migrations remain separate destructive actions rather than automatic plan steps; an operator must arrange a database backup and recovery procedure before running them.

`setup.sh` detects supported Debian/Ubuntu versions, installs prerequisites and CloudPanel when absent, publishes a complete non-root-readable Node.js distribution in `/usr/local/lib/panelavo-node`, and uses `npx` with the pinned pnpm version so Corepack is optional. It creates the panel site/user, deploys and builds the repository, configures narrow sudo access, starts PM2, and configures firewall rules and SSL. Provisioning leaves fail2ban's `sshd` jail active; an optional detected-client exemption is temporary and removed on exit. It recovers SSH connection metadata through process ancestry and preserves the live SSH port in UFW. Existing active UFW installations are updated; inactive UFW installations receive prepared rules but are activated only through the explicit `ENABLE_UFW=true` opt-in.
