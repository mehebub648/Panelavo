# Architecture

Panelavo is a single Next.js App Router application deployed as a CloudPanel Node.js site. Nginx/CloudPanel proxies the panel domain to the single PM2-managed Next.js process on port 10443.

Browser requests enter pages and route handlers in `src/app`. Shared validation lives in `src/schemas`; UI components live in `src/components`. Protected server routes obtain opaque application sessions from `src/server/auth`, revalidate current CloudPanel identity and roles, and call the live adapter in `src/server/cloudpanel`. The adapter invokes `/usr/bin/clpctl` for supported mutations and the allow-listed local `scripts/cloudpanel-bridge.php` for Symfony/Doctrine reads and narrowly mapped site-management actions that the public CLI does not expose. CloudPanel remains authoritative for users, roles, MFA, sites, assignments, site types, runtimes, roots, and upstream settings.

The bridge also executes allow-listed Git operations as each site's Unix user. Git is non-interactive, uses that user's home and deployment key, records new SSH host keys with OpenSSH's `accept-new` policy, and is bounded by a five-minute timeout. Clones are restricted to empty configured site roots so existing application files cannot be overwritten, with one exception: an ACME-managed `.well-known` directory is preserved while the repository is checked out.

File-manager uploads use base64 JSON and are limited to 64 MiB decoded at both browser and bridge boundaries. The panel Nginx vhost accepts request bodies up to 96 MiB to cover encoding and JSON overhead; `setup.sh` owns that proxy directive and applies it with validation and rollback.

The Super Admin update API reads a configurable public HTTPS Git repository from the panel settings store. Checks use `git ls-remote`; installs launch the non-root `scripts/self-update.sh` worker. The worker holds a `.data` lock, shallow-clones `main`, verifies the Panelavo package contract, installs from the frozen pnpm lockfile, and completes a production build in staging before synchronizing the live tree. While the persisted state is queued, updating, or reloading, the shared panel layout blocks interaction and authenticated APIs reject normal operations, while server-rendered panel pages remain available behind the overlay. It excludes `.data` and `.env.local`, records durable status and logs, then cleans staging and records the old process PID before asking PM2 to reload. Because PM2 kills the worker with the old process tree, the replacement Panelavo process owns completion of this PID-bound handoff. Legacy states whose installed and remote commits already match are recovered automatically, including removal of their stale update lock. The repository is a trusted application-code source but receives no root execution path.

Application-owned metadata and encrypted credentials live under `.data` in the deployed site. Host configuration comes from `.env.local`; secrets must not reach client components. A single process is required by the current session and in-process rate-limit design.

## Managed website Operations

The CloudPanel site type, including Panelavo's Docker overlay for CloudPanel reverse-proxy records, is authoritative. `scripts/cloudpanel-bridge.php` inspects only the configured application root and returns structured architecture evidence, manifests, declared scripts, installed tool versions, permissions, PM2 state, Compose capabilities, and validation results. `src/server/sites/site-operations.ts` converts that report into a primary architecture, alternatives, explicit preflight checks, action groups, and a recommended server-owned plan. A manifest can be detected while every related action remains blocked because its runtime, configuration, permission, or safety check failed.

The adapters cover npm, pnpm, Yarn, and Bun lockfile selection; Composer with Laravel and WordPress actions; Python through uv, Poetry, Pipenv, or pip plus Django actions; direct static sites; reverse-proxy checks; PM2 process management; and Docker Compose. The configured root is not searched recursively. A JavaScript workspace therefore needs usable root scripts or explicit root-level configuration, and Panelavo does not guess a generated static output directory or change the CloudPanel document root automatically.

The browser submits only a validated action identifier plus a narrowly validated script or process name when required, or a known deployment-plan identifier. The server expands these into fixed executable argument arrays, a fixed working directory, bounded time and output, and no shell. A per-site operation lock rejects overlapping runs, plans stop after the first failed step, and the UI keeps missing tools and other blockers visible instead of attempting fallback installation.

Rootful Docker Compose is a separate host-risk boundary. Only a Super Admin may execute it, using the preflight-selected Compose file and stable project name explicitly. The bridge requires the Docker CLI, Compose v2 plugin, reachable daemon, valid resolved configuration, and a passing host-safety policy before any root command. Panelavo does not install Docker, alter the daemon, or add the panel/site user to the root-equivalent `docker` group.

Operations currently run synchronously within one bounded HTTP request and mutate the configured live application root. They do not create release directories, atomically switch releases, or provide automatic code or database rollback. Laravel and Django migrations remain separate destructive actions rather than automatic plan steps; an operator must arrange a database backup and recovery procedure before running them.

`setup.sh` detects supported Debian/Ubuntu versions, installs prerequisites and CloudPanel when absent, publishes a complete non-root-readable Node.js distribution in `/usr/local/lib/panelavo-node`, and uses `npx` with the pinned pnpm version so Corepack is optional. It creates the panel site/user, deploys and builds the repository, configures narrow sudo access, starts PM2, and configures firewall rules and SSL. Provisioning leaves fail2ban's `sshd` jail active; an optional detected-client exemption is temporary and removed on exit. It recovers SSH connection metadata through process ancestry and preserves the live SSH port in UFW. Existing active UFW installations are updated; inactive UFW installations receive prepared rules but are activated only through the explicit `ENABLE_UFW=true` opt-in.
